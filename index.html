Blob Soft Delete & Versioning
  • Blob soft delete: Deleted blobs retained for a configurable period (1–365 days) — recoverable in the portal or via API.
  • Container soft delete: Same concept at container level.
  • Blob versioning: Every write creates a new version. Previous versions are accessible and can be restored. Works alongside soft delete.
  • Immutability (WORM): Time-based retention policies lock blobs — no delete or overwrite, even by admins. Used for compliance (SEC 17a-4, FINRA).

  • Components: Storage Sync Service → Sync Group → Server Endpoint (on-prem) + Cloud Endpoint (Azure file share).
  • Supports multi-site sync — same share accessed from multiple on-prem locations and Azure simultaneously.

Network Security Groups (NSG)

NSGs contain inbound and outbound security rules. Each rule has: priority (100–4096, lower = processed first), source/destination (IP, CIDR, service tag, ASG), port, protocol (TCP/UDP/ICMP/Any), and action (Allow/Deny). The first matching rule wins — Azure stops evaluating further rules.

Application Security Groups (ASG): Tag VMs by workload (e.g. "WebTier", "DBTier") and use those tags in NSG rules instead of explicit IP addresses. Rules stay valid as VMs scale in/out.
VNet Peering
  • Regional peering: VNets in the same Azure region. Low latency, no bandwidth limits, traffic stays on Microsoft backbone.
  • Global peering: VNets in different Azure regions. Same mechanism, minor performance difference.
  • Non-transitive: If VNet A peers with B, and B peers with C — A and C cannot reach each other through B. Requires explicit A↔C peering (or a hub with routing).
  • No overlapping address spaces — peered VNets must have unique CIDR ranges. Cannot be changed post-peering.
  • Peering is bidirectional but each direction must be explicitly created (both A→B and B→A).

  • Replication: Continuously replicate Azure VMs to a secondary region. RPO ~30 seconds.

AZ-104 Practice Questions

15 scenario-based questions. Work through each one before revealing the answer.

15 questions · Click each question to reveal the answer and explanation
1 A company needs to prevent all Azure administrators from accidentally deleting a critical production storage account, while still allowing configuration changes. Which resource lock type should you apply? ▼
Answer
CanNotDelete lock.

CanNotDelete prevents deletion of the resource by any user — even Owners — but still allows read and write operations (configuration changes, data uploads, etc.). ReadOnly would also prevent the deletion, but it blocks all write operations too, which would prevent legitimate configuration changes. Since the requirement is to allow changes but prevent deletion, CanNotDelete is the correct and least-restrictive option.
Rule: CanNotDelete = block delete only. ReadOnly = block delete AND modify.
2 Your organisation wants all newly created Azure resources to automatically have a "CostCenter" tag applied. Resources missing the tag must be auto-remediated without blocking creation. Which Azure Policy effect should you use? ▼
Answer
Modify.

The Modify effect adds, updates, or removes tags on resources at create or update time. When paired with a remediation task, it also fixes existing non-compliant resources. The key word "auto-remediated without blocking" eliminates Deny (which blocks creation). Append can add properties but cannot modify existing tag values. DeployIfNotExists deploys a related resource — it is not used for tag enforcement.
Rule: Auto-add or fix tags without blocking = Modify. Block creation = Deny.
3 A compliance team requires financial records stored in Azure Blob Storage to be retained for 7 years with no possibility of deletion or modification, even by administrators. Which feature enforces this? ▼
Answer
Immutability policy (time-based retention — WORM).

A time-based WORM (Write Once Read Many) immutability policy locks blobs for a defined retention period. No user — including global admins — can delete or overwrite blobs during this period. This satisfies SEC 17a-4, FINRA, and similar compliance requirements. Soft delete only protects for 14 days (configurable max 365). Resource locks protect at the resource level, not the data level. Lifecycle management automates tiering but doesn't prevent deletion.
Rule: Immutable data retention for compliance = WORM immutability policy, not soft delete or lifecycle management.
4 A company has 5 Azure subscriptions across 3 business units. They want to apply a common set of Azure Policy assignments and RBAC roles to all subscriptions with the least administrative effort. What should they implement? ▼
Answer
Management Groups.

Management Groups provide a scope above subscriptions. Policy and RBAC assignments at the Management Group level automatically cascade to all child subscriptions, resource groups, and resources. Without management groups, each subscription would require individual assignment — 5× the management overhead. Management groups support a maximum of 6 levels of depth below the tenant root.
Rule: Apply policy/RBAC to multiple subscriptions at once = Management Groups.
5 An Azure VM needs to authenticate to Azure Key Vault to retrieve secrets at startup. You must avoid storing credentials in the VM or in code. What is the recommended approach? ▼
Answer
System-assigned Managed Identity on the VM, with Key Vault access granted via RBAC (Key Vault Secrets User role).

Managed Identity provides an automatically managed Azure AD identity for the VM. Azure handles credential rotation transparently. The VM can obtain a token from the Instance Metadata Service (IMDS) without storing any credentials. Assign the Key Vault Secrets User role to the managed identity at the Key Vault scope. No secrets are ever stored in code or config files.
Rule: No credentials in code + Azure service access = Managed Identity.
6 A development team wants to deploy and test a new version of their App Service web application without taking production offline. They need to verify the new version works before making it live. Which feature enables this? ▼
Answer
Deployment slots (requires Standard tier or higher).

Deploy the new version to a staging slot, run smoke tests, then swap staging into production with zero downtime. The swap is instant — Azure routes traffic to the new slot while the old slot remains available for rollback. App settings marked as "slot settings" stay on their configured slot and don't swap with the app — useful for environment-specific connection strings. Requires Standard tier (5 slots) or Premium (20 slots).
Rule: Zero-downtime deployment with rollback capability = Deployment Slots.
7 A storage account holds 50 TB of sensor logs. Logs older than 30 days are rarely accessed; logs older than 90 days are never accessed. You must minimise storage costs automatically. What should you configure? ▼
Answer
Azure Blob Storage lifecycle management policy.

Create a rule that transitions blobs to Cool tier after 30 days of last modification, and to Archive tier after 90 days. Optionally, add a delete action after the compliance retention period. Lifecycle management policies run daily and automatically move blobs between tiers based on age or last access time — no manual intervention. This is the cost-efficient alternative to keeping all data in Hot tier.
Rule: Automatic cost-based blob tier transitions = Lifecycle management policy.
8 An administrator cannot connect from a VM in VNet-A to a VM in VNet-C. VNet-A is peered with VNet-B, and VNet-B is peered with VNet-C. All peerings show as Connected. What is the issue? ▼
Answer
VNet peering is non-transitive. A→B and B→C does not create A→C connectivity.

Each peering is a direct, point-to-point connection. Traffic from A cannot transit through B to reach C — Azure does not route peering traffic transitively. To connect A and C, you must either: (1) create an explicit A↔C peering, or (2) deploy a hub VNet with a Network Virtual Appliance (NVA) or Azure Firewall and use UDRs to route A→hub→C.
Rule: VNet peering is non-transitive. Explicit peering or hub-and-spoke routing required for full mesh.
9 An NSG has these inbound rules: Priority 200 – Allow TCP 443 from Any; Priority 300 – Deny All from Any. A request arrives on port 80. What happens? ▼
Answer
The traffic is denied by the default DenyAllInBound rule (priority 65500).

NSG rules are processed in priority order (lowest number first). Priority 200 allows TCP 443 only — port 80 doesn't match. Priority 300 denies all — port 80 matches here, so traffic is denied. (Even if Priority 300 wasn't present, the built-in DenyAllInBound at 65500 would deny it.) The key concept: the first matching rule wins. Port 80 doesn't match rule 200, so processing continues to rule 300 which denies it.
Rule: NSG processes rules lowest priority number first. First match wins — stops evaluating.
10 An administrator needs to verify why an Azure VM cannot make an outbound connection to a specific external IP on port 443. The VM has an NSG applied. Which Network Watcher tool should they use? ▼
Answer
IP Flow Verify.

IP Flow Verify checks whether a specific traffic flow (source IP, destination IP, port, protocol, direction) is allowed or denied by the NSG rules applied to a VM. It instantly shows which NSG rule is responsible for allowing or blocking the traffic — without needing to manually parse all NSG rules. Next Hop shows the routing path, not NSG evaluation. Connection Monitor provides ongoing monitoring, not one-time diagnostics.
Rule: "Is this traffic blocked by NSG?" = IP Flow Verify. "Where does this traffic go?" = Next Hop.
11 All traffic from a subnet to the internet must be inspected by an Azure Firewall deployed in a hub VNet. The spoke subnet is peered to the hub. Currently traffic bypasses the firewall and goes directly to the internet. What must you configure? ▼
Answer
User Defined Route (UDR) on the spoke subnet with a default route (0.0.0.0/0) pointing to the Azure Firewall's private IP address as the next hop.

By default, Azure routes traffic between peered VNets directly and sends internet traffic out the system default gateway — it does not automatically route through the hub firewall. A UDR overrides system routing. Setting 0.0.0.0/0 → Virtual Appliance (Firewall private IP) forces all outbound traffic through the firewall. NSGs filter traffic but cannot change the routing path.
Rule: Force traffic through firewall/NVA = UDR. NSGs cannot change routing paths.
12 A company needs to back up 50 Azure VMs and run application-consistent backups of SQL Server running inside one of those VMs. Which vault type supports both workloads? ▼
Answer
Recovery Services Vault.

Recovery Services Vault supports Azure VMs, SQL Server in VMs, Azure Files, and SAP HANA in VMs. Backup Vault supports Azure Managed Disks, Azure Blobs, and Azure Database for PostgreSQL — it does not support Azure VMs or SQL in VMs. Since both workloads (VMs + SQL in VM) require Recovery Services Vault, that is the only option that satisfies the full requirement.
Rule: Azure VMs + SQL in VM = Recovery Services Vault. Managed Disks + Blobs = Backup Vault.
13 An administrator accidentally deleted several critical files from an Azure Files share 2 days ago. The share has soft delete enabled with a 7-day retention. Can the files be recovered, and how? ▼
Answer
Yes — recover the soft-deleted share in the Azure portal or via Azure CLI/PowerShell.

Soft delete retains deleted Azure Files shares (and their contents) for the configured retention period (1–365 days). At 2 days, the files are well within the 7-day window and are in a "soft deleted" state. Navigate to Storage Account → File shares → show deleted shares → select the share → Undelete. The share and all its files are restored to the state at the time of deletion. After the retention period, the data is permanently purged.
Rule: Soft delete = recoverable within retention window. Always verify soft delete is enabled on production shares.
14 A company's EU data residency policy requires that all Azure Storage data must remain within the EU region. Availability Zone resilience is also required. Which redundancy option should you choose? ▼
Answer
ZRS (Zone-Redundant Storage).

Regional compliance immediately eliminates GRS, GZRS, RA-GRS, and RA-GZRS — all replicate data to a secondary Azure region outside the EU. Remaining options: LRS (single datacenter, no zone resilience) and ZRS (3 zones within one region). Since AZ resilience is required, ZRS is the answer. ZRS replicates 3 copies across 3 physically separate datacenters within the same region — surviving a datacenter failure without leaving the region.
Rule: Regional compliance → eliminate all geo-redundant options. Zone HA needed → ZRS over LRS.
15 After deploying a new Azure VM, the team needs to automatically install IIS and configure a custom website without logging into the VM. The configuration must run immediately post-deployment. What is the recommended approach? ▼
Answer
Custom Script Extension (CSE).

The Custom Script Extension downloads and executes a script (PowerShell for Windows, Bash for Linux) on the VM post-deployment. It can install IIS (Install-WindowsFeature -Name Web-Server), copy website content from Azure Blob Storage, and configure settings — all without RDP/SSH access. CSE can be applied at deployment time via ARM template, Azure CLI, or the portal. Desired State Configuration (DSC) is another option but requires a DSC configuration file; CSE is simpler for straightforward install-and-configure tasks.
Rule: Post-deployment automated script on a VM = Custom Script Extension.

Single region or multi-region?
│
├── SINGLE REGION
│   ↓
│   HTTP/HTTPS?
│   ├── YES → Application Gateway (L7, URL routing, WAF, session affinity)
│   └── NO  → Azure Load Balancer (L4, TCP/UDP, internal or public)
│
└── MULTI-REGION
    ↓
    HTTP/HTTPS + single hostname?
    ├── YES → Azure Front Door
    └── NO  → Azure Traffic Manager (any protocol, DNS-based)

Framework 4 — Storage Redundancy

Is there a regional data-residency compliance requirement?
(data must stay within one Azure region)
│
├── YES → Eliminate GRS, GZRS, RA-GRS, RA-GZRS (all replicate to secondary region)
│         Remaining options: LRS or ZRS
│         ↓
│         Need to survive datacenter/zone failure?
│         ├── YES → ZRS (3 zones in region, survives datacenter failure)
│         └── NO  → LRS (cheapest, single datacenter)
│
└── NO  → All options eligible
          ↓
          Need read access to secondary at all times (not just after failover)?
          ├── YES → RA-GRS or RA-GZRS
          └── NO
              ↓
              Need zone-level fault tolerance in primary AND geo-redundancy?
              ├── YES → GZRS (or RA-GZRS for read access)
              └── NO  → GRS (cheaper, one datacenter in primary region)

Framework 5 — Azure Disk Tier Selection

Start with stated IOPS requirement:
│
├── ≤ ~2,000 IOPS → Standard HDD sufficient (dev/test only)
├── ≤ ~6,000 IOPS → Standard SSD sufficient
├── ≤ ~20,000 IOPS → Premium SSD sufficient ← most common exam answer
└── > 20,000 IOPS (up to 160,000) → Ultra Disk required

THEN apply throughput check on the surviving tier.
THEN apply cost constraint: lowest tier clearing ALL gates = answer.
Trap: Ultra Disk is never the answer when Premium SSD meets the stated thresholds — cost minimisation overrides a desire for headroom.

Framework 6 — Azure Files: Standard vs Premium

Does the workload require NFS protocol?
├── YES → Premium (FileStorage account) — NFS is Premium-only
└── NO (SMB or REST only)
    ↓
    Does the compliance/DR requirement need geo-redundancy (GZRS)?
    ├── YES → Standard (GPv2) — GZRS only available on Standard
    └── NO  → Either tier eligible; Premium for IOPS-sensitive, Standard for cost

Framework 7 — On-Premises Connectivity

Primary requirement?
│
├── MINIMISE LATENCY / guaranteed bandwidth
│   → ExpressRoute (private circuit, off public internet, up to 100 Gbps)
│   ⚠ NOT encrypted by default — add IPSec over ER if needed
│
├── ENCRYPTED tunnel over public internet (acceptable latency)
│   ├── Site-to-site → VPN Gateway S2S
│   └── Remote users → VPN Gateway P2S
│
├── Branch offices: automated connectivity + central security + backbone
│   → Virtual WAN Standard
│
└── Secure browser-based RDP/SSH without public IP
    → Azure Bastion

Framework 8 — Messaging Service Selection

What is the consumption pattern?
│
├── Single consumer processes each message
│   ├── Need reliability, ordering, dead-letter, sessions → Service Bus Queue
│   └── Cost over features → Azure Storage Queue
│
├── MULTIPLE consumers each need their OWN COPY → Service Bus Topics + Subscriptions
│
├── High-volume event STREAMING (telemetry, logs, clickstream)
│   → Event Hubs (consumer groups, replay, Capture → Blob/ADLS)
│
└── Reactive event routing (trigger Functions, Logic Apps on resource events)
    → Event Grid (push, at-least-once, low latency)

Framework 9 — Compute Service Selection

Is it HPC / large-scale parallel workload (rendering, simulation, batch jobs)?
├── YES → Azure Batch (no time limit, managed nodes, auto-scale)
└── NO
    ↓
    Is execution time ≤ 10 min (default) or ≤ 60 min (max) and infrequent?
    ├── YES + HTTP trigger + minimise cost → Azure Functions (Consumption plan)
    └── NO
        ↓
        Is it containerised without orchestration?
        ├── YES + shared storage across containers → ACI + Azure Files volume
        ├── YES + short-lived, isolated → ACI
        └── NO
            ↓
            Is it microservices / container orchestration at scale?
            ├── YES → AKS
            └── NO
                ↓
                Web app or REST API (no containers)?
                ├── YES → App Service
                └── Full OS control / lift-and-shift → Virtual Machine

Framework 10 — Availability / Business Continuity Scope

What failure must the solution survive?
│
├── Hardware component / rack / server failure
│   (datacenter failure acceptable)
│   → Availability Set  [SLA 99.95%]
│     Fault domains: up to 3 (separate racks/power)
│     Update domains: 5–20 (rolling patching)
│
├── Datacenter / power outage affecting a physical location
│   (regional failure acceptable)
│   → Availability Zone  [SLA 99.99%]
│     3 physically separate datacenters within region
│     Cross-zone latency: <2ms
│
└── Regional failure (entire Azure region down)
    → Multi-region architecture
      ├── IaaS: Azure Site Recovery (RPO ~30s, RTO minutes)
      └── PaaS: SQL geo-replication / auto-failover groups
               Cosmos DB multi-region writes
               Azure Storage RA-GRS/RA-GZRS
